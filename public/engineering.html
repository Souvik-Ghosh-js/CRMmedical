§ 01

The brief

An enterprise pharmaceutical ERP, engineered to compete with the category leaders.

The mandate was a life-sciences ERP in the class of SAP for Life Sciences and Oracle Health Sciences — 27 module families spanning inventory, GST accounting, warehouse and cold-chain, manufacturing execution, quality management, regulatory affairs, pharmacovigilance, serialization, distributor management, and AI-assisted analytics. Every screen production-grade, with the columns and controls that real pharmaceutical operations depend on.

Rather than build 45 screens as 45 one-off pages, we invested first in the substrate: a reactive data layer, a config-driven module engine, and a shared component system. That substrate is what let breadth and consistency arrive together — every module inherits the same behavior, and new ones are declared, not hand-coded.

The strategy Build the abstractions before the features. A reactive store, one CRUD engine, and a design system mean the 45th screen costs a fraction of the first — and behaves identically.
§ 02

Architecture: a reactive data layer behind a stable interface

Every screen talks to data through one API-shaped module, never to storage directly.

The cornerstone is a reactive data layer that presents each entity as a collection with a clean, API-shaped surface. Components never reach into storage themselves; they read and write through hooks, and a publish/subscribe layer re-renders exactly the views affected by a change. Because the entire application depends only on this interface — not its implementation — the data tier can evolve independently of the 45 screens above it.

// the data contract every screen is written against
export function list(name) { ... }         // SELECT *
export function insert(name, record) { ... } // create + notify subscribers
export function update(name, id, patch) { ... }
export function remove(name, id) { ... }
export function subscribe(name, fn) { ... }   // pub/sub → reactive re-render
export function nextNumber(name) { ... }     // INV-2026-0001 document sequences

Reactivity is intentionally lightweight — a registry of listeners keyed by collection, and a useCollection(name) hook that re-renders subscribers on write. No heavyweight state framework; the flow is direct and easy to reason about, which matters when 45 screens share it.

§ 03

FEFO: the domain logic at the core of stock

In pharmacy you don't ship oldest-in — you ship first-to-expire. That rule drives inventory everywhere.

First-Expiry-First-Out is the beating heart of pharmaceutical stock control. When an invoice line sells 8 units, the system doesn't just decrement a count — it walks that product's batches ordered by expiry date, consuming the shortest-dated first, and records exactly which batches each line drew from.

// allocate a sale across batches, earliest expiry first
export function deductFEFO(productId, qty) {
  let remaining = qty; const allocations = []
  for (const b of fefoBatches(productId)) {   // sorted by expiryDate ASC
    if (remaining <= 0) break
    const take = Math.min(remaining, b.qty)
    update('batches', b.id, { qty: b.qty - take })
    allocations.push({ batchId: b.id, batchNo: b.batchNo, qty: take })
    remaining -= take
  }
  return { allocations, shortfall: remaining }
}

Those allocations aren't cosmetic — they're what makes the 1-day invoice edit fully reversible. Editing an invoice calls restoreAllocations() to return every unit to its original batch, then re-runs FEFO on the new quantities. Stock is conserved exactly: restore 8 → batch returns to full, re-deduct 12 → spills correctly into the next batch, and the group totals reconcile.

Engineered for correctness Batch allocations turn "edit an invoice" from a risky mutation into a precise restore-and-reapply. The audit trail records the revision; the ledger never drifts.
§ 04

One engine, nineteen modules

The config-driven CRUD component that made enterprise breadth affordable.

Nineteen enterprise modules — WMS, QMS, MES, Regulatory, Pharmacovigilance, Serialization, Cold Chain and the rest — don't each hand-roll a table and a form. They declare a config object and hand it to one CrudModule engine that renders the list, search, filters, summary stats, add/edit/delete workflow, Excel export, audit logging, and pagination.

// Qms.jsx — an entire quality-management screen, declaratively
const config = {
  collection: 'qmsRecords', title: 'Quality Management',
  numberPrefix: 'DEV',                // auto DEV-0001 on create
  columns: [ { key: 'status', badge: (r) => [r.status==='Closed'?'green':'amber', r.status] }, ... ],
  fields:  [ { key: 'capaType', type: 'select', options: [...] }, ... ],
  stats:   (rows) => [ { label: 'Open', value: rows.filter(...).length } ],
}
export default () => <CrudModule config={config} />

The payoff compounds. When pagination was added later, it went into CrudModule once and all nineteen module tables gained it instantly. Domain-accurate columns — Cold Chain tracks mean-kinetic-temperature and excursion duration; WMS tracks bin, zone, RFID tag and cycle-count variance — live in the config, not in bespoke JSX.

§ 05

Multi-company consolidation

Three group companies, one consolidated view — with inter-company eliminations done right.

The platform runs a multi-entity group: a parent pharmacy plus distribution and labs companies in different states, each with its own GSTIN. Every transaction carries its owning company, so consolidation is real arithmetic, not a mock-up. The engine sums per-company revenue, COGS, GST and collections, then eliminates inter-company transfers — stock transfers, service charges and management fees — so intra-group activity doesn't double-count.

The subtlety worth calling out: fund transfers and loans between entities net to zero across the group and are shown for reconciliation only, while stock-and-service transfers are eliminated from both revenue and COGS. A guard clamps eliminations so a stray entry can never drive consolidated revenue below the group's real earnings — a rule verified with a standalone numeric simulation before it shipped.

Correctness by construction Consolidated statements are only trustworthy if eliminations are principled. We modeled which transaction types eliminate against which line, then proved the math holds under simulation.
§ 06

Charts and pagination, built from scratch

No charting library, no table library — a ~150kB gzipped bundle for the whole platform.

The analytics are pure SVG. Catmull-Rom smoothing turns data points into flowing cubic Béziers, with gradient area fills, dashed gridlines, hover tooltips, and a niceMax() axis-rounding helper so the Y-axis lands on clean values. Five chart types — line, bar, donut, sparkline — in one dependency-free module, driving the dashboard, MIS, and BI views.

Pagination followed the same instinct: a usePagination(rows, size) hook returning the current slice plus controls, and a <Pagination> footer with a windowed page list (1 … 5 [6] 7 … 47). Totals and exports stay bound to the full filtered set — you paginate the view, never the arithmetic.

§ 07

Where parallel orchestration earned its keep

Two jobs were embarrassingly parallel, so they ran in parallel.

Building nineteen module screens, and later retrofitting pagination onto nine hand-built tables, are both wide, independent, mechanical jobs. Each unit is self-contained — ideal for fanning out the work against a strict contract, then integrating and verifying centrally.

The discipline that made it safe: write the shared engine first, so every unit extends one convention instead of inventing nineteen; define a strict output contract; then reconcile centrally — wire routes, load data, and run a real production build as the honest gate. The pagination pass returned nine edited files; a single build confirmed all nine integrated without breakage.

The pattern Scaffold the shared abstraction → fan out against a contract → integrate and verify with a real build. Parallelism buys speed only when the seams are defined before the work starts.
§ 08

Realistic operating history, by design

A live system is judged on populated dashboards — so the platform ships with a believable three-month history.

A dedicated generator produces roughly three months of operating data — hundreds of invoices weighted toward the parent company, purchases every few days, receipts against credit customers, stock movements, and credit/debit notes — all with correct tax math, batch allocations, and inter-state IGST flags. Everything downstream lights up immediately with data that reconciles: dashboard graphs, MIS, GST reporting, receivables ageing, and the trial balance.

A SCHEMA VERSION constant governs it. Bump the version and the dataset regenerates on next load, so schema changes never leave anyone staring at stale figures. The generator was validated with a standalone numeric simulation before being trusted in the app — the same rigor applied to the consolidation engine.

Takeaway Realistic, self-consistent history is a feature. It's the difference between a system that looks live and one that merely renders.
